여기서 말하는 AI 냄새 제거는 “AI를 안 쓴 척하는 위장”이 아닙니다.
정확히는 지나치게 균일하고 기계적으로 정돈된 표면을 걷어내고, 실제 도메인 판단·운영 감각·트레이드오프가 코드 밖이 아니라 코드 안에서 보이게 만드는 작업입니다.

즉, 코드를 일부러 거칠게 만드는 게 아니라,

죽은 추상화를 없애고
상태와 책임의 축을 분리하고
generic한 이름을 도메인 이름으로 바꾸고
검증 로직을 순수한 규칙으로 분리하고
“왜 이렇게 했는가”를 타입과 주석과 테스트 이름으로 드러내는 것

이 핵심입니다.

이번 새 버전에서 그 작업이 가장 필요한 곳은 크게 여섯 군데입니다.

ReplyAdmissionService.kt + ReplyService.kt
CommandIngressService.kt
ReplyRoutes.kt + RequestAuthenticator.kt
AppRuntime.kt + 일부 운영/설정 계층
tools/iris-ctl의 reply_modal.rs, main.rs, messages.rs, members.rs
tools/iris-daemon의 config_sync.rs, process.rs

아래부터는 “어떻게 분해할지”와 “왜 그게 AI 냄새를 줄이는지”를 새 버전 기준으로 정확하게 정리하겠습니다.

1. 먼저, 새 버전에서 보이는 AI 냄새의 정체부터 분명히 하겠습니다

이 새 버전은 예전보다 훨씬 좋아졌습니다.
특히 production ForTest seam이 많이 줄었고, 정책과 보안 쪽은 상당히 단단합니다.

그럼에도 아직 남아 있는 “기계적이고 인위적인 냄새”는 대체로 네 종류입니다.

첫째, 큰 파일이 너무 “정돈된 채로” 큽니다

대표적으로:

tools/iris-ctl/src/views/reply_modal.rs
app/src/main/java/party/qwer/iris/ingress/CommandIngressService.kt
app/src/main/java/party/qwer/iris/reply/ReplyAdmissionService.kt
tools/iris-ctl/src/main.rs
tools/iris-ctl/src/views/members.rs
tools/iris-ctl/src/views/messages.rs

이 파일들은 지저분해서 큰 게 아니라, 질서정연하게 큰 상태입니다.
이게 전형적으로 “한 컨텍스트 안에 기능을 계속 쌓아 올린” 느낌을 줍니다.

둘째, generic한 이름이 아직 조금 남아 있습니다

예를 들어:

PipelineRequest
DispatchRequest
DispatchResult
CommandIngressDispatchConfig
ReplySendLane

이 중 일부는 의미가 약합니다.
예를 들어 ReplySendLane은 이름은 도메인적이지만, 실제로는 queue lane을 실질적으로 바꾸지 않습니다.
이런 개념은 “모델은 있는데 체온이 없는 상태”를 만듭니다.

셋째, 서로 다른 상태 축이 한 객체에 섞여 있습니다

가장 대표적인 것이 reply_modal.rs입니다.

field focus
overlay selector 상태
form draft
validation 결과
sending 상태
derived state (is_open_chat)

이게 한 struct 안에 다 들어 있습니다.

이런 구조는 기능은 되지만, 코드가 숨을 쉬지 못합니다.
사람이 이해할 때는 “지금 이 상태는 폼의 상태인가, 화면의 상태인가, 파생 상태인가?”를 계속 물어야 하기 때문입니다.

넷째, 테스트 중 일부가 “동작”보다 “구조”를 따라갑니다

예를 들면 tools/iris-ctl/src/main.rs 테스트 중
include_str!("main.rs")로 소스 텍스트를 읽어서 특정 문자열이 들어 있는지 검사하는 테스트는, 아이디어는 나쁘지 않지만 상당히 기계적입니다.

이런 테스트는 “왜 이 기능이 중요한가”보다
“현재 이 파일이 이런 문자열을 포함하는가”를 확인합니다.
AI가 증폭시킨 테스트에서 자주 나오는 패턴입니다.

2. 분해의 원칙부터 잡아야 합니다

여기서 가장 중요한 건 무작정 많이 쪼개지 않는 것입니다.

AI 냄새를 지운다고 해서 파일을 20개씩 나누면 오히려 더 AI스럽습니다.
왜냐하면 미세하게 잘게 쪼개진 generic helper 천국도 AI 냄새의 한 형태이기 때문입니다.

분해는 아래 기준으로 해야 합니다.

기준 1. 상태를 누가 소유하는가

상태의 주인이 다르면 파일도 나눕니다.

예:

queue/worker 상태
UI overlay 상태
form draft 상태
replay buffer 상태
config persistence 상태
기준 2. 어떤 invariant를 지키는가

같은 invariant를 지키는 코드끼리는 같이 두고,
다른 invariant를 지키는 코드는 나눕니다.

예:

ReplyRoutes.kt에서 multipart metadata-before-image 규칙
RequestAuthenticator.kt에서 nonce/body-hash 순서 규칙
CommandIngressService.kt에서 commit prefix 전진 규칙
기준 3. pure rule과 effectful orchestration을 분리합니다

이건 매우 중요합니다.

validation
selection rule
classifier
projection
formatting

이런 것은 가능한 한 pure하게 분리합니다.

반대로

network
file IO
DB polling
channel/worker lifecycle
UI mutation

은 orchestration 레이어에 둡니다.

기준 4. generic helper보다 domain object를 우선합니다

예를 들어 cachedResolve() 같은 generic helper는 재사용 면에서는 좋지만,
코드에서 무슨 일을 하는지 안 보입니다.

그럴 때는 차라리:

SenderNameResolver
RoomMetadataResolver
IngressBacklog
ValidatedReplyTarget

처럼 도메인 객체로 승격하는 편이 훨씬 좋습니다.

3. ReplyAdmissionService.kt와 ReplyService.kt는 같이 정리해야 합니다

이 영역은 새 버전에서 가장 먼저 손봐야 하는 서버 쪽 대상입니다.

현재 구조의 문제

ReplyAdmissionService.kt 안에는 지금 다음이 모두 있습니다.

PipelineRequest
worker 상태 모델
admission command 모델
worker launch loop
worker registry
enqueue retry/replace
graceful shutdown
debug snapshot

즉, 서비스가 아니라 작은 런타임입니다.

그리고 ReplyService.kt는 다시

façade
nested pipeline class
status transition processor
image handle cleanup

을 같이 들고 있습니다.

이 둘이 합쳐지면 “의도는 좋은데 덩어리가 크고 조금 기계적인” 인상이 납니다.

어떻게 나눌 것인가

권장 구조는 이렇습니다.

이 이름 변경만으로도 코드가 훨씬 덜 generic해집니다.

왜 이 분해가 AI 냄새를 줄이는가

지금은 “모든 것이 reply admission service 안에서 일어난다”는 느낌입니다.
이건 깔끔하지만 생기가 없습니다. 이유는 어떤 개념이 어디까지 자기 책임인지 코드가 말하지 않기 때문입니다.

분해 후에는 이렇게 읽혀야 합니다.

ReplyAdmissionService는 actor façade
ReplyLaneCoordinator는 registry와 상태 전이 소유
ReplyLaneWorker는 worker 하나의 lifecycle 소유
ReplyLaneJobProcessor는 status transition과 send sequencing 소유

이렇게 되면 복잡성이 사라지는 것이 아니라, 복잡성의 위치가 설명 가능해집니다.
그게 사람 코드의 생기입니다.

특히 꼭 고쳐야 할 세부 사항
ReplySendLane는 정리하는 편이 맞습니다

현재 ReplyService.kt에서 lane을 받지만 실제 효과가 약합니다.
의미 없는 개념은 AI 냄새를 강하게 만듭니다.

둘 중 하나를 택해야 합니다.

정말 queue partition에 영향이 없다면 삭제
정말 lane이 필요하다면 ReplyQueueKey와 worker 배치까지 영향이 가게 만들기

지금 코드 기준으로는 삭제 쪽이 더 맞아 보입니다.

rejection cleanup은 admission이 소유하게 바꾸십시오

지금은 ReplyService.enqueueRequest()가 거절 시 pipelineRequest.discard()를 호출합니다.

이건 아래로 옮기는 것이 맞습니다.

is ReplyAdmissionCommand.Enqueue -> {
    val result = coordinator.enqueue(command.key, command.job)
    if (result.status != ReplyAdmissionStatus.ACCEPTED) {
        runCatching { command.job.abort() }
    }
    command.reply.complete(result)
}

이렇게 해야 admission layer가 진짜 owner가 됩니다.

shutdownSuspend()는 actor 자체도 정리해야 합니다

지금은 worker는 닫지만 commands, commandScope는 남습니다.
TERMINATED라면 actor infra도 같이 닫는 편이 맞습니다.

4. CommandIngressService.kt는 “좋은 코드인데 너무 많은 것을 안다”가 핵심입니다

이 파일은 새 버전에서 가장 인간적인 파일 중 하나입니다.
동시에 가장 과밀한 파일이기도 합니다.

지금 구조는 “정말 잘 돌아가지만, 사람이 문제를 어디까지 나눠 생각했는지 표면에 안 보이는 구조”입니다.

이건 코드에 생기를 넣는 대표적인 방법입니다.
왜냐하면 코드를 읽는 사람이 “아, 이 사람은 이 문제를 이렇게 나눠 생각했구나”를 느낄 수 있기 때문입니다.

꼭 같이 고쳐야 하는 세부 사항

작지만 lifecycle이 더 정확해집니다.

현재 섞여 있는 것

왜 이 분해가 AI 냄새를 줄이는가

여기에 꼭 추가하면 좋은 것

예를 들어 multipart 경로 위에 아래 정도는 넣는 것이 좋습니다.

/**

이건 설명이 아니라 판단 기록입니다.
바로 이런 것이 코드의 생기입니다.

14. 실행 순서는 이렇게 잡는 것이 가장 좋습니다
1단계

가장 먼저 표면 냄새가 큰 곳을 정리합니다.

reply_modal.rs 모듈 분리
main.rs에서 source text test 제거/대체
messages.rs, members.rs projection 분리

이 단계는 체감 효과가 큽니다.

2단계

ReplyAdmissionService.kt + ReplyService.kt
CommandIngressService.kt
ReplyRoutes.kt

이 단계에서 코드베이스의 목소리가 바뀝니다.

3단계

운영 레이어를 정리합니다.

AppRuntime.kt → RunningRuntime + StartupBuilder

4단계

남은 actor/policy 계층을 정리합니다.

새 버전 기준으로 AI 냄새를 지우는 가장 좋은 방법은
코드를 더 인간처럼 어설프게 만드는 것이 아니라,
코드가 이미 품고 있는 사람의 판단을 더 선명하게 드러내는 것입니다.

핵심만 다시 압축하면 이렇습니다.

ReplyAdmissionService.kt와 ReplyService.kt는 façade / coordinator / worker / job / processor로 나눕니다.
CommandIngressService.kt는 backlog / decode / routing assembly / resolvers로 나눕니다.
ReplyRoutes.kt는 route 파일로 줄이고 collector / stager / target validator를 밖으로 뺍니다.
reply_modal.rs는 draft / UI state / overlay / validate / render / input으로 찢습니다.
main.rs의 소스 텍스트 검사형 테스트는 없애고 계약 테스트로 바꿉니다.
AppRuntime.kt는 lateinit 조립식에서 running graph + rollback builder로 올립니다.
config_sync.rs, process.rs는 문자열 중심 구현에서 정책 객체 중심 구현으로 올립니다.
전체적으로는 generic 이름을 줄이고, invariant comment와 decision-oriented log를 늘립니다.

이렇게 가면 이 코드는 “매끈하고 기계적인 코드”에서
도메인과 운영 판단이 숨 쉬는 코드로 확실히 바뀝니다.
